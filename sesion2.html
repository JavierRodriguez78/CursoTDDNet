Test Driven Development

  • El TDD es una metodología recomendada para los procesos de desarrollo ágil
  • Se usan test unitarios
  • El ciclo de TDD consiste en:
    • Traducimos los requisitos a pruebas (test)
    • Escibimos los test antes que el código
    • Resolvemos el test con el mínimo de código necesario
      • Recordemos el principio KISS (Keep It Simple, Stupid)
    • Refactorizamos después de cada iteraccion (si es necesario)
  • Se suele combinar con otras prácticas ágiles como el pair programming

Ciclo del TDD

  • RED
    • Escribimos un test y los dejamos en rojo esto significa que el test falle.
      • El test tine que FALLAR, no dar error.
  • GREEN
    • Escribimos la mínima cantidad de código que resuelva el test
      • Se trata de solucionar SOLO ese test. Principio KISS.
      • No debemos adelantar acontecimientos o presuponer futuras necesidades
  • BLUE
    • Analizamos nuestro código y buscamos posibles mejoras.
      • Renombrando variables/métodos
      • Eliminación de smells
      • Eliminación de duplicidades

Ventajas del TDD

  • La calidad del software aumenta.
  • Conseguimos código altamente reutilizable.
  • El trabajo en equipo se hace más fácil, une a las personas. Nos permite onfiar en nuestros compañeros aunque tenga menos experiencia.
  • Multiplica la comunicación entre los miembros del equipo.
  • Las personas encargadas de la garantía de calidad adquieren un rol más intelegente e interesante.
  • Escribir el ejemplo(test) antes que el código nos obliga a escribir el mínimo de funcionalidad necesaria evitando sobrediseñar.
  • Cuando revisamos un proyecto desarrollado mediante TDD, nos damos cuenta de que elos test son la mejor documentacion técnica que podemos consultar a la hora de enteder qué misión cumple cada pieza del puzzle.

Limitaciones del TDD

  • Interfaces gráficas de usuario
    • No existe manera fácil de testearlas.
    • Hay otras metodologías quizá mejor orientadas a este tipo de testing,como el BDD
  • Base de datos
    • Es complicado de testear el código que accede a base de datos.
    • Se neesita tener una base de datos conocida y rsetear su estado antes de cada test
